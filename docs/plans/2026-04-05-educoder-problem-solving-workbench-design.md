# Educoder 做题工作台（极简做题流）设计文档

**日期：** 2026-04-05  
**分支：** `feature/task8-local-judge`  
**状态：** 结构瘦身方案与交互规则已定稿（以 2026-04-06 最终确认为准）  
**目标：** 把扩展从“同步/资料工具”收敛成“极简做题工作台”：用户一键同步整章后，直接在 VS Code 里看题、写代码、看测试、看答案、跑本地、再提交头哥；不再暴露 README、报告 JSON、索引 JSON、内部元数据等无助于做题的产物。

---

## 0. 2026-04-06 最终确认（权威覆盖规则）

> **如果本文较早段落的示例、目录名或措辞与本节冲突，一律以本节为准。**

### 0.1 用户可见目录最终形态

用户可见目录收敛为：

```text
problem/
code/
tests/
answers/
```

说明：

- `code/current`、`tests/all`、`answers/unlocked` 这类命名只允许作为内部迁移/兼容实现细节存在；
- 在用户界面、Explorer 体验、文档说明中，都必须统一成 `code / tests / answers` 的做题语言；
- 模板代码、已通过代码、报告 JSON、manifest、metadata、recovery 等全部属于内部实现层。

### 0.2 工作区默认挂载章节根

同步完成后，VS Code 默认接入 **当前章节根**，而不是整个 `Educoder Local OJ` 总根。

目标：

- 降低 Explorer 层级深度；
- 用户打开后就聚焦当前这一章；
- 减少“为什么树这么深”的产品负担。

### 0.3 一键同步完成后的默认落点

同步完成后默认打开逻辑固定为：

1. 优先打开 **未完成 / 资料异常 / 资料不完整** 的题；
2. 若都正常，则打开 **上次打开的题**；
3. 若没有历史记录，再打开 **第一题**。

不再简单粗暴地永远打开第一题。

### 0.4 多答案默认行为

若某题存在多份答案：

- 默认直接打开第一份答案；
- 不先打开答案目录；
- 暂不弹选择器；
- 后续如确有必要，再追加轻量选择能力。

### 0.5 高级能力的产品位置

高级能力从普通用户侧栏界面中移除，但仍保留在 **命令面板** 中，供调试或高级用户使用。

这意味着：

- 侧栏不再显示强制提交、回滚、同步远端仓库、对比模板等；
- 但这些能力不是立即物理删除，而是退回非主路径入口。

### 0.6 部分题失败时整章同步不应中断

若某题缺题面 / 测试 / 答案：

- 整章同步继续进行；
- 该题标记为 **资料不完整** 或 **同步异常**；
- Task Tree 中应有明确状态；
- 不能因为单题失败拖死整章同步。

### 0.7 提示策略

正常交互以 **侧栏摘要** 为主，toast 只保留极少数必要场景：

- 同步完成
- 致命错误
- 明确的用户动作确认

run / submit / open 等常规反馈不应被一堆弹窗打断。

### 0.8 Task Tree 的显示语言

Task Tree 中：

- 只显示干净名称；
- 不显示 `[id]`；
- 状态用轻量图标 / 小点 / 徽标表达；
- 不让树节点看起来像调试数据结构。

### 0.9 磁盘目录名继续保留 ID

磁盘真实目录继续保留 `[id]`，保证稳定性与可恢复性；

但：

- Task Tree/UI 隐藏 ID；
- 用户主路径不依赖 ID 进行识别。

### 0.10 不额外生成给人看的报告文件

本轮不生成 `report.md` / `report.html` 之类的额外产物。

结论：

- 用户看摘要区；
- 程序读内部 JSON；
- 不新增一套“人类报告文件”维护负担。

### 0.11 Explorer 默认隐藏内部内容

Explorer 默认隐藏：

- `_educoder/`
- `.vscode/`
- `code/template/`
- `code/passed/`
- `*.manifest.json`
- metadata / recovery / judge json 等机器文件

用户默认只看做题材料，不看内部实现层。

### 0.12 标题区信息量保持克制

当前题标题区统一采用：

- 简洁题目标题
- 短 breadcrumb

例如：

```text
2-4 · 第1关：链表操作
```

不展示冗长的课程/章节/作业/任务完整路径。

## 1. 问题确认

当前扩展虽然已经具备整章同步、题面落盘、答案拉取、本地评测、提交头哥等能力，但产品体验仍然带有明显的“内部工具感”：

- 页面上堆了过多按钮，主次关系混乱；
- `打开测试集` / `打开答案` 仍然可能打开索引文档，而不是用户真正想看的内容；
- `README.md`、`reports/*.json`、`tests/index.json`、`answers/index.md` 等文件暴露给用户，徒增噪音；
- `资料完整度 / 材料导航 / 开始做题 / 答案与解析 / 进阶工具` 这类分区工程味太重，不像“做题产品”；
- 用户真正只关心：题面、代码、测试、答案、最近结果、运行本地、提交评测。

因此，本轮不再把重点放在“补更多入口”，而是放在：

> **删除无用概念、收缩文件结构、弱化内部实现、强化主做题路径。**

---

## 2. 产品目标

用户对产品的正确心智模型应当是：

1. 粘贴一个章节 URL；
2. 点击 `一键同步本章`；
3. 自动拉齐本章所有关卡；
4. 左侧选题；
5. 自动打开题面和当前代码；
6. 需要时直接看测试、看答案；
7. 点击 `运行本地测试`；
8. 通过后点击 `提交评测`。

这意味着：

- **不要让用户理解“同步目录”**；
- **不要让用户理解 README / manifest / metadata / recovery / report JSON**；
- **不要让用户自己找测试目录、答案目录、当前代码目录**；
- **不要把高级调试能力和主路径动作放在同一层级。**

---

## 3. 核心设计决策

### 3.1 主路径只保留“做题五件套”

用户可见、可理解、可直接操作的核心对象收敛为：

- 题面
- 当前代码
- 测试
- 答案
- 评测结果

其中主动作收敛为：

- `打开题面`
- `打开代码`
- `运行本地测试`
- `提交评测`

测试/答案作为次级入口保留；高级动作折叠隐藏。

### 3.2 删除 README 作为产品路径

`README.md` 只是重复解释目录结构，不帮助用户做题，也会让文件树显得更杂。

结论：

- 不再在 UI 中显示 `打开 README`
- 不再生成 `README.md`

### 3.3 不再把报告 JSON 当成用户入口

`reports/latest_local.json`、`reports/latest_submit.json` 这类文件适合程序读，不适合用户看。

结论：

- UI 不再显示 `打开报告`
- 用户直接在当前题工作台里看到最近测试摘要
- 如需落盘结构化结果，放到 `_educoder/` 内部目录，不作为用户主路径暴露

### 3.4 测试 / 答案入口必须直达真实材料

用户点击：

- `打开测试集` 时，应直接定位 `tests/all/`
- `打开答案` 时，应直接打开第一份已解锁答案正文；若无正文，则 reveal `answers/unlocked/`

不能再优先打开：

- `tests/index.json`
- `answers/index.md`

### 3.5 用户可见文件树必须极简

每道题默认暴露给用户的内容尽量收敛成：

```text
problem/
  statement.md
code/
  current/
tests/
  all/
answers/
  unlocked/
```

如存在样例，可放在：

```text
problem/samples/
```

以下内容不应继续作为用户主视图的一部分：

- `README.md`
- `reports/`
- `tests/index.json`
- `answers/index.md`
- `problem/metadata.json`
- `answer_info.json`
- `code/template/`
- `code/passed/`
- 其他 manifest / recovery / sync JSON

### 3.6 内部实现统一下沉到 `_educoder/`

程序仍然可能需要保留：

- 任务元数据
- 恢复状态
- 答案元信息
- 原始接口快照
- 本地/远端评测结构化结果
- 模板代码 / 已通过代码 / 远端仓库镜像

这些都应该统一作为内部实现数据放在 `_educoder/` 下，而不是污染用户做题视图。

### 3.7 当前题 UI 只做“做题动作编排”，不再做按钮堆砌

当前题卡片应改成：

1. 标题区：当前题名 + 轻量状态摘要；
2. 主操作区：`打开题面 / 打开代码 / 运行本地测试 / 提交评测`；
3. 次级入口：`测试 / 答案`；
4. 最近结果：直接展示通过数、失败数、首个失败用例；
5. 高级动作：折叠收起。

`资料完整度`、`打开 README`、`打开报告`、大段 `进阶工具` 不应再出现在主视图。

---

## 4. 信息架构

### 4.1 Activity Bar

仍保留两个核心视图：

1. **题目目录（Task Tree）**
   - 课程 / 章节 / 作业 / 题目树
   - 当前题高亮
   - 支持筛选
   - 题目状态简洁展示

2. **当前题目（Current Task）**
   - 当前题标题
   - 主做题动作
   - 最近结果摘要
   - 次级入口（测试 / 答案）
   - 折叠的高级操作

### 4.2 Explorer

Explorer 只承担“自然浏览真实做题材料”的职责。

目标体验不是让用户读内部工程目录，而是让用户一眼看到：

- `problem/statement.md`
- `code/current/`
- `tests/all/`
- `answers/unlocked/`

### 4.3 编辑器

打开题目后自动打开：

- `problem/statement.md`
- `code/current/<主编辑文件>`

这样用户无需再问“我接下来点哪个”。

---

## 5. 用户旅程（更新版）

### 5.1 首次进入

用户看到的首页应尽可能简单：

- 主按钮：`一键同步本章`
- 次按钮：`更换存放目录`
- 说明文案：粘贴章节链接后自动拉整章题目包

### 5.2 同步完成

同步完成后：

1. 自动定位到首题；
2. Explorer 中能看到真实做题材料；
3. 自动打开题面和当前代码；
4. 当前题卡片只展示做题所需动作。

### 5.3 做题中

用户的主要流转应当是：

- Task Tree 切题
- 编辑器写代码
- 需要时看测试/答案
- 点击本地测试
- 失败时直接看失败摘要
- 点击提交评测

这里不要求用户手动打开 JSON 报告，也不要求用户理解目录元数据。

## 6. 非目标

本轮明确不做：

- Webview 内重造 IDE
- Webview 内重造完整文件树
- 面向用户暴露内部 JSON 状态系统
- 通过 README 解释目录结构
- 通过 JSON 报告作为主反馈方式

---

## 7. 技术方向约束

### 7.1 一键同步主路径不变

继续采用：

- `syncCollectionPackages(...)` 作为公开入口
- `syncCurrentCollection(...)` 作为内部步骤

### 7.2 当前代码 / 测试 / 答案仍坚持真实落盘

即便 UI 收缩，也必须继续满足：

- 真实文件生成
- 真实命令注册/调用
- 真实本地评测结果落盘（内部目录）
- 真实工作区接入

### 7.3 评测结果改为“侧栏摘要 + 内部落盘”双轨

用户层：
- 看摘要
- 看失败 case
- 触发下一步动作

程序层：
- 保留结构化结果，供恢复/状态模型/提交流使用

## 8. 验收标准（更新版）

本轮设计以以下结果为准：

1. UI 不再出现 `README` 和 `报告 JSON` 作为主路径入口；
2. `打开测试集` 不再打开 `tests/index.json`；
3. `打开答案` 不再打开 `answers/index.md`；
4. 用户可见文件树明显收缩为做题材料；
5. 当前题 UI 不再堆叠大量同级按钮；
6. 本地测试结果可直接在当前题区域读懂，而不是依赖打开 JSON；
7. 所有内部实现数据统一弱化到 `_educoder/`。

## 9. 已确认的交互方向（第一批）

### 9.1 当前题卡片采用 A+B 混合版

本轮已确认：当前题区域不再使用“多分区 + 按钮墙”形式，而改成 **极简做题卡片**。

卡片结构固定为：

1. **题目标题区**
   - 当前题名称
   - 可附一行轻量状态，如“测试 6/8 · 首个失败 case_007”

2. **最近结果摘要**
   - 用一句话直接展示最近一次本地评测或提交结果
   - 不要求用户再去打开 JSON 报告

3. **四个主按钮**
   - `打开题面`
   - `打开代码`
   - `运行本地测试`
   - `提交评测`

4. **两个次级入口**
   - `测试`
   - `答案`

5. **更多操作（折叠）**
   - 强制提交
   - 重新拉答案
   - 对比模板 / 对比答案
   - 其他高级动作

### 9.2 明确不再采用的旧布局

当前这种布局在产品上视为废弃：

- `资料完整度` 大段展示
- `材料导航` + `开始做题` + `答案与解析` + `进阶工具` 四五块并列
- README / 报告 / 元数据同级暴露
- 多个高级操作和主做题动作并排出现

### 9.3 视觉层级原则

已确认采用以下层级：

- **主按钮**：高强调，仅 4 个
- **次级入口**：弱强调，小按钮或链接式入口
- **高级动作**：默认收起，不抢主路径
- **结果摘要**：放在主按钮上方，作为当前题的第一反馈信息

## 9. 待二次讨论后补充的内容

以下属于下一轮“交互感受优化”议题，待讨论后再追加进计划：

- 当前题卡片的更具体视觉排版；
- 主按钮/次按钮/链接按钮的样式层级；
- 最近结果摘要的具体展示形式；
- 失败用例快捷入口是否直接做成显式按钮；
- 更像 VS Code 原生体验的紧凑布局与文案细化。

## 10. 已确认的交互细节（第二批）

### 10.1 当前题卡片必须首屏无滚动

当前题区域必须控制在单屏内完成主要信息表达，避免再次退化成“长滚动按钮墙”。

因此当前题卡片只允许包含：

- 一行题目标题
- 两行结果摘要（头哥 / 本地）
- 四个主按钮
- 一行次级文本入口

不再允许把 `资料完整度`、`材料导航`、`开始做题`、`答案与解析`、`进阶工具` 等多个区块堆叠在首屏内。

### 10.2 切题后的打开策略固定

当用户在 Task Tree 里切换题目时，统一采用以下行为：

1. `problem/statement.md` 以 **preview** 方式打开；
2. `code/current/<主编辑文件>` 以正常编辑页打开；
3. 最终焦点停留在代码编辑器；
4. Explorer reveal 当前题或当前代码位置。

目标是让用户“知道切到哪题了”，但不打断立即写代码。

### 10.3 当前题卡片只保留四个主按钮

主按钮固定为：

- `打开题面`
- `打开代码`
- `运行测试`
- `提交评测`

按钮文案必须短，不再出现长工程文案，例如：

- `打开当前代码`
- `提交到头哥（先跑本地）`
- `仅提交到头哥（高级）`

复杂规则改为辅助说明，不再写进按钮文案。

### 10.4 次级入口统一改成文本链接式

以下入口不再做成按钮，而改成轻量文本入口：

- `测试集`
- `答案`
- `失败输入`
- `失败输出`

其中：

- `测试集` 常驻显示；
- `答案` 常驻显示；
- `失败输入 / 失败输出` 仅在最近一次本地测试失败时显示。

这样既保留快速定位能力，又不和主按钮抢视觉层级。

### 10.5 不再在普通用户 UI 中显示“更多操作”

本轮确认：

- 当前题卡片中不再显示 `更多操作`
- 不再在普通用户主界面暴露强制提交、对比模板、回滚模板、同步远端仓库等工程/调试能力

如这些能力仍需保留，只能退回命令面板、调试模式或其他非主路径入口。

### 10.6 结果摘要改成两行，并带轻量状态色

当前题卡片顶部统一显示两行摘要：

1. **头哥结果**
2. **本地结果**

推荐文案示例：

- 头哥：`未提交`
- 头哥：`提交中…`
- 头哥：`已通过 8/8`
- 头哥：`未通过 6/8`
- 本地：`未运行`
- 本地：`通过 8/8`
- 本地：`编译失败：main.cpp 第 17 行附近`
- 本地：`失败 case_007：期望 5，实际 4`

视觉上采用轻量状态色或小图标：

- 灰：未运行 / 未提交
- 黄：提交中 / 本地通过待提交
- 绿：通过
- 红：失败 / 编译错误

### 10.7 运行测试的反馈必须明显但不打断编辑

点击 `运行测试` 后：

- 不自动乱开文件；
- 不打断当前代码编辑焦点；
- 通过刷新摘要区反馈结果；
- 若失败，再额外显示 `失败输入 / 失败输出` 文本入口。

核心原则：**反馈要明显，但不能破坏写代码节奏。**

### 10.8 提交评测采用“摘要主导、少弹窗、不打断”策略

点击 `提交评测` 后：

1. 自动保存；
2. 自动跑本地；
3. 本地不过则停止，不提交头哥；
4. 本地通过才进入远端提交。

UI 反馈以摘要区为主：

- `本地未通过，未提交头哥`
- `头哥提交中…`
- `头哥已通过 8/8`
- `头哥未通过 6/8`

尽量避免连续弹出消息框打断用户。

### 10.9 答案是必备材料，不存在“未解锁答案”的产品概念

本轮明确确认：

- 一键同步本章时，每道题的答案都必须被拉取；
- 对用户而言，不存在“未解锁答案”这个产品概念；
- 若答案缺失，属于同步异常，而不是正常空态；
- UI / 目录 / 文案中都不应再暴露 `unlock`、`unlocked`、`answer_info` 这类实现词汇。

这意味着用户可见目录命名也要同步收敛：

```text
answers/
```

而不是：

```text
answers/unlocked/
```

内部实现需要的答案元数据仍然可以保留在 `_educoder/answers/` 下。

### 10.10 用户语言必须替代工程语言

本轮确认所有用户可见元素都必须使用“做题语言”，而不是“实现语言”。

不应出现在用户主视图中的词包括：

- unlocked
- metadata
- manifest
- recovery
- report json
- answer_info

用户看到的应该只有：

- 题面
- 代码
- 测试
- 答案
- 本地结果
- 头哥结果



### 10.11 最新补充：2026-04-06 的 Explorer / Task Tree / 默认落点规则

- Explorer 默认隐藏内部实现内容；
- Task Tree 只显示用户名称，不显示 ID；
- 磁盘继续保留 ID；
- 同步完成后不再机械打开第一题，而是优先打开异常题 / 上次题 / 第一题；
- 默认工作区挂载章节根；
- 不额外生成给人看的报告文件。
